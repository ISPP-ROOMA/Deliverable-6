User: Quiero crear una actualización en la que se mida el Cycle Time (el tiempo que se tarda en cerrar una issue desde que se asigna a la persona). Hasta ahora en Calculos de tiempo se ha medido el Lead Time (el tiempo que se tarda en cerrar issues desde que se abren).

Para esta actualizacion es crucial que se llame Lead Time al Lead Time en lugar de duration, ya que vamos a añadir las gráficas de Cycle Time por persona con fechas de inicio y fin igual que en Lead Time. 

Además, es crucial crear otra gráfica que sea el cálculo de rendimiento. Deberia ser una pestaña mas dentro del desplegable github y es basicamente una nueva gráfica que sea el cálculo entre dos fechas de los story points completados por persona entre el cycle time de la forma sum(sp) / sum(cycle_time). El cycle time debe ser repartido entre las personas a las que se encargó la issue.

Utiliza si te es necesaria esta informacion ademas de integrarla por completo en la aplicación (incluso colores y personalizacion de colores)

ENDPOINTS A USAR
----------------
 
1. Issues cerradas
   GET /repos/{owner}/{repo}/issues?state=closed
   Campos: created_at, closed_at, assignees, labels, milestone, number
 
2. Fecha real de asignación + movimiento de columnas
   GET /repos/{owner}/{repo}/issues/{n}/timeline
   - Evento "assigned"              -> created_at del evento + assignee.login
   - Evento "moved_columns_in_project" -> columna origen/destino + created_at

   Cycle time real (desde asignación)
  assigned_at = primer evento 'assigned' del timeline para esa persona
  cycle_time  = closed_at - assigned_at   (días)
  velocidad   = story_points / cycle_time (SP por día)
  Requiere una llamada extra al timeline por issue.
  Es la métrica más justa: elimina el tiempo en backlog.

  EADERS NECESARIOS PARA LA API
--------------------------------
Authorization: Bearer {TOKEN}

X-GitHub-Api-Version: 2022-11-28
 
Para project classic, añadir también:
Accept: application/vnd.github.starfox-preview+json
(necesario para obtener el campo project_card en los eventos)
 
 
CONSIDERACIONES DE IMPLEMENTACIÓN
------------------------------------
- La API pagina de 30 en 30 por defecto
  -> iterar con ?per_page=100&page=N hasta que la respuesta esté vacía
 
- El timeline es una llamada por issue
  -> gestionar rate limit (5000 req/h con token autenticado)
  -> implementar caché local o flag para no repetir llamadas ya hechas
 
- Una issue puede tener múltiples assignees y reasignaciones
  -> opción simple: atribuir al assignee en el momento del cierre
  -> opción precisa: usar todos los eventos assigned/unassigned del timeline
     para calcular cuánto tiempo la tuvo cada persona y repartir el SP
 
- Issues sin label sp:x
  -> excluirlas del cálculo de velocidad pero contarlas como throughput
 
- Filtrar pull requests (los PRs también son issues en la API)
  -> filtrar con: "pull_request" not in issue

## Plan: Cycle Time y Rendimiento GitHub

Se va a ampliar el análisis de GitHub para separar claramente Lead Time y Cycle Time, renombrar por completo el uso actual de duration a Lead Time, y añadir una nueva pestaña de Rendimiento con la métrica por persona: sum(story_points)/sum(cycle_time). El diseño reutiliza el pipeline actual de jobs, gráficas, personalización y regeneración, incorporando timeline por issue con reparto preciso por asignación real.

**Steps**
1. Fase 1 - Contrato de datos y naming global (bloquea el resto).
1. Definir métricas canónicas en todo el flujo: lead_time_days, cycle_time_days y performance_sp_per_day.
2. Establecer compatibilidad para jobs antiguos (que solo tienen duration) sin romper detalle/regeneración/histórico.
3. Alinear nombres técnicos y visibles para eliminar duration en UI, CSV, chart keys y etiquetas.

2. Fase 2 - Fetch de GitHub y cálculo Cycle Time real (depende de fase 1).
1. Extender fetch de issues cerradas con cabeceras API requeridas y paginación explícita per_page=100, page=N.
2. Añadir llamada por issue al timeline y parsear eventos assigned/unassigned por persona.
3. Construir tramos temporales por assignee y calcular cycle time distribuido por tiempo real.
4. Mantener exclusión de PRs (pull_request), solo issues cerradas para cycle/performance.
5. Implementar control de rate limit y caché local por issue para no repetir timelines ya consultados.

3. Fase 3 - Agregaciones y exportación (depende de fase 2).
1. Reemplazar agregados avg_duration/total_duration por avg_lead_time/total_lead_time.
2. Añadir agregados avg_cycle_time/total_cycle_time por persona.
3. Calcular rendimiento por persona: sum(sp_share)/sum(cycle_time_share).
4. Excluir issues sin sp:x del rendimiento, pero conservarlas para throughput.
